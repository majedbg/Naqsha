# R2.5 Review — Phase 2.5 Admin Panel (Security + a11y)

**Reviewer:** R2.5 (adversarial, read-only)
**Scope:** spec §8 admin panel — `TopNav.jsx`, `AdminPage.jsx`, `OrgLauncher.jsx`, route wiring in `App.jsx`.
**Mandate:** UI gating is NOT the boundary — RLS is. Verify both; flag UI that *implies* protection the DB lacks.

## KEY QUESTION
> Is org creation / admin-assignment reachable by a non-platform-admin through any path (UI OR service), or is RLS the firm boundary?

**ANSWER:** Org creation / admin-assignment is **NOT reachable by a non-platform-admin through any path.** RLS is the firm boundary.
- *UI path:* the create form and the per-org assign form (OrgRow) are rendered ONLY when `isPlatformAdmin()===true`. A non-platform user with no admin orgs sees Access-denied; an org-admin sees OrgLauncher (read-only). Neither renders any write control.
- *Forced-render / service path:* both writes go exclusively through `createOrg` / `assignOrgAdmin` → Supabase → RLS `is_platform_admin()` (verified in R2). There is no client-only "trust me" insert in any of the four files. Even if a non-admin force-mounted the form, the INSERT is denied server-side.
- *Gate fidelity:* `isPlatformAdmin()` is DB-backed (`platform_admins` filtered by `auth.email()`), not a client guess; the `.eq('email', user.email)` filter is load-bearing.
- *Defense-in-depth:* the `/admin` route mounts AdminPage directly and AdminPage self-gates — it does NOT rely on the TopNav tab being hidden.

The blocking issue below is an **error-handling / a11y defect, not a boundary breach**: an `assignOrgAdmin` failure (reachable only by a platform admin) fails silently with no announced error.

## Test verification
All four target suites run green: **21/21 passed** (`TopNav` 4, `AdminPage` 5, `OrgLauncher` 5 incl. router, `App` 7). Note: 21/21 green is NOT evidence against the HIGH finding — there is no test exercising `assignOrgAdmin` failure, which is exactly what a silent-failure path produces.

## Findings

### `src/components/nav/TopNav.jsx` (+ TopNav.test.jsx)

**Security**
- VERIFIED-SAFE — The Admin tab is purely a *convenience*. Visibility = `isPlatformAdmin() || listMyAdminOrgs(userId).length>0`. `isPlatformAdmin()` (platformService.js:16-29) queries `platform_admins` filtered by `auth.email()`, mirroring the RLS policy — it reflects the DB, not a client guess. `listMyAdminOrgs` (membershipService.js:64-74) is RLS-scoped by `user_id`. The tab renders NO write controls — it is a `<Link to="/admin">` only. Hiding the tab does not gate anything; the gate lives at the route + AdminPage (see App.jsx / AdminPage findings).
- NIT — TopNav.jsx:14 — `isPlatformAdmin()` is called even when `user` is null/anon. It does `supabase.auth.getUser()` and returns false for no email, so it is *safe* (no leak) — just a wasted round-trip on every nav mount for anon users. No security impact. Fix (optional): short-circuit when `!userId`.
- LOW — TopNav.jsx:14 — the `Promise.all` has no `.catch`. If `isPlatformAdmin()` throws (it re-throws Supabase errors, platformService.js:27), the rejection is unhandled and `showAdmin` stays false. Fail-closed (correct security posture) but produces an unhandled promise rejection. Fix: add `.catch(() => alive && setShowAdmin(false))`.

**A11y**
- VERIFIED-SAFE — `<nav aria-label="Primary">` (line 26) is a labeled landmark; Admin is a real `<Link>` (focusable, keyboard-reachable). Test asserts `role="link"` href `/admin` and the labeled `role="navigation"`.

**Proven vs assumed:** Tab visibility for all four cohorts (platform admin / org admin / plain-anon / brand-link) is PROVEN by tests. The non-admin negative is robust (waits for both gate calls before asserting absence). The error/no-catch path is UNTESTED (assumed).

### `src/pages/AdminPage.jsx` (+ AdminPage.test.jsx)

**Security**
- VERIFIED-SAFE (route gate / defense-in-depth) — AdminPage self-gates: it does NOT trust the TopNav tab. On mount it calls `isPlatformAdmin()` (DB-backed). While resolving, `isAdmin===null` renders `null` (line 100-102) — no form leaks during the async window. Only `isAdmin===true` renders the Organizations create/assign UI (line 117+). A non-platform user with zero admin orgs gets `Access denied` (line 110-114) and the create form is NEVER mounted. PROVEN by test "shows access-denied … and never exposes the create form" (asserts no name field, no create button, AND `createOrg`/`listOrgs` not called).
- VERIFIED-SAFE (no client-only write path) — both writes go through the RLS service: `handleCreate`→`createOrg` (line 55), `handleAssign`→`assignOrgAdmin` (line 17). There is no direct `supabase.from('orgs').insert` in the component. Even if a non-admin force-rendered the form, the INSERT hits Supabase RLS `is_platform_admin()` (verified in R2) and is denied server-side. **The UI gate is convenience; RLS is the boundary.**
- VERIFIED-SAFE (no cross-org leak in non-platform branch) — for `!isAdmin && hasAdminOrgs`, AdminPage renders `<OrgLauncher />`, which lists only `listMyAdminOrgs(userId)` (RLS-scoped by `user_id`). No other org's data is fetched in this branch. `listOrgs()` (all-orgs) is called ONLY inside the `admin===true` branch (line 77).
- VERIFIED-SAFE (errors surface, not silent) — `handleCreate` wraps `createOrg` in try/catch and sets `error` → `role="alert"` (line 144). PROVEN by test "surfaces a createOrg error in a role=alert" — so an RLS denial for a forced-UI non-admin surfaces as a visible error, not silent success.
- HIGH — AdminPage.jsx:15-19 — `OrgRow.handleAssign` (assign-admin) has **NO try/catch**. `assignOrgAdmin` re-throws on RLS denial / duplicate / network error (platformService.js:53). Unlike `handleCreate`, a failure here produces an unhandled promise rejection and `setEmail("")` never runs, but critically **no error is shown to the user** — the form silently appears to do nothing on failure. This violates spec/finding-5 intent ("failures surface as errors, not silent success") for the assign path specifically. NOTE: this is reachable only by a *platform admin* (the only cohort that sees OrgRow), so it is not a privilege-escalation bug — it is a silent-failure / error-handling defect. The assign-error path is also UNTESTED (the happy-path test mocks a resolved value). Fix: mirror `handleCreate` — wrap in try/catch and render a `role="alert"` in the row on failure.
- LOW — AdminPage.jsx:92-94 — the `isPlatformAdmin()` chain's `.catch` sets `isAdmin=false`, which for a user with no admin-org info falls through to `Access denied`. Fail-closed (correct). But a *platform admin* hitting a transient network error on `isPlatformAdmin()` is shown "Access denied" with no retry/error distinction. Acceptable for MVP (fail-closed > fail-open); noted for UX, not security.

**A11y**
- VERIFIED-SAFE — create-form inputs are wrapped in `<label>` (Name/Slug/Accent/Logo), associated by nesting; tests resolve them via `getByLabelText`. Assign input has `<label>Admin email</label>` nesting. Submit buttons are real `<button type="submit">`. The error is `<p role="alert">` (announced). Section is `aria-label="Organizations"` with an `<h1>`. Access-denied is also `role="alert"`.
- LOW (a11y) — AdminPage.jsx:27, 129-141 — `<input>`s have no `id`/`htmlFor`; association relies on DOM nesting inside `<label>`. Works for AT and RTL `getByLabelText`, but explicit `id`+`htmlFor` is more robust. NIT.

**Proven vs assumed:** Platform-admin happy path (list/create/assign), non-admin access-denied (incl. service-not-called), and the **createOrg** error→alert are PROVEN. The **assignOrgAdmin error path** is ASSUMED/UNTESTED and is in fact a silent-failure defect (HIGH above). The `isAdmin===null` no-flash window and the `hasAdminOrgs`→OrgLauncher branch are not directly tested in AdminPage.test (relies on OrgLauncher's own tests).

### `src/components/admin/OrgLauncher.jsx` (+ OrgLauncher.test.jsx)

**Security**
- VERIFIED-SAFE (no cross-org leak) — OrgLauncher's ONLY data source is `listMyAdminOrgs(userId)` (line 16), RLS-scoped by `user_id`/`is_admin`/`status='active'` (membershipService.js:64-74). It renders links to `/o/${slug}/admin` for the caller's own admin orgs only. `userId` defaults to `user?.id` from `useAuth` (line 9). PROVEN: tests assert the exact orgs returned, an empty state with NO links, and `listMyAdminOrgs` called with the resolved id. No path renders another user's orgs.
- VERIFIED-SAFE (no writes) — OrgLauncher is read-only navigation; it issues no create/assign call. No privilege surface.

**A11y**
- VERIFIED-SAFE — org links are real `<Link>` elements (focusable, keyboard-reachable), under an `<h2>Your organizations</h2>` heading. States (loading/empty/list) are text. Tests resolve links via `getByRole('link')`.
- LOW (a11y) — OrgLauncher.jsx:39 — the error state is a plain `<p>` without `role="alert"`, so a screen-reader user who is focused elsewhere when the async fetch fails is NOT notified of the failure. Contrast with AdminPage's create error which IS `role="alert"`. Inconsistent. Not a security issue. Fix: add `role="alert"` to the error `<p>`.

**Proven vs assumed:** All four states (list / empty / loading / error) and the auth-id fallback are PROVEN. RLS scoping is enforced at the service (verified R2), not re-asserted here — correct.

### `src/App.jsx` (route wiring) (+ App.test.jsx)

**Security (defense-in-depth — the critical check)**
- VERIFIED-SAFE — `/admin` routes DIRECTLY to `<AdminPage />` (line 20). The route does NOT depend on the TopNav tab being visible. A user navigating directly to `/admin` (typing the URL, bookmark, link) still mounts AdminPage, which independently calls `isPlatformAdmin()` and renders Access-denied for non-admins. **Route-gating does NOT rely solely on the hidden tab** — the tab (TopNav) and the page gate (AdminPage) are independent; the page gate is the client-side gate, RLS is the real boundary. App.test proves `/admin`→AdminPage routing; AdminPage.test proves the gate. Correct separation.
- VERIFIED-SAFE — `<TopNav>` mounts above `<Routes>` inside `<AuthProvider>` (line 14-15), matching spec §8 ("wraps the router"). Persistence proven by App.test ("keeps TopNav persistent on /admin").
- OBSERVATION (not a finding) — there is no `<ProtectedRoute>` wrapper; each protected page self-gates. This is an acceptable pattern here because (a) the client gate is explicitly NOT the security boundary and (b) every write is RLS-protected server-side. If a future page forgets to self-gate, there is no central guard to catch it — worth a note for maintainers, but not a defect for the four MVP targets.

| HIGH | AdminPage.jsx:15-19 | `assignOrgAdmin` has no try/catch — RLS denial / dup / network failure is silent (no `role=alert`), unlike createOrg. Mandate-#5 item fails for the assign path. Platform-admin-only (not a privilege bug); RLS boundary intact. |
| LOW | OrgLauncher.jsx:39 | Error state is a plain `<p>` without `role="alert"` — fetch failure not announced. Inconsistent with AdminPage. |

| LOW (a11y) | AdminPage.jsx:27,129-141 | Inputs use label-nesting, no explicit `id`/`htmlFor`. Works; more robust with explicit association. |
| LOW | TopNav.jsx:14 | `Promise.all` has no `.catch` → unhandled rejection on gate error (fails closed). |
| NIT | TopNav.jsx:14 | `isPlatformAdmin()` round-trips even for anon users. No security impact. |

**Verified-SAFE (key):** route gates independently of the tab; both writes are RLS-only with no client insert; `isPlatformAdmin()` is DB-backed; OrgLauncher leaks no cross-org data; createOrg error surfaces as `role=alert`; nav/forms are labeled and keyboard-reachable.

## Verdict: FINDINGS-MUST-FIX
The security boundary is sound — **RLS is firm and no non-platform-admin write path exists.** The blocker is the **HIGH** silent-failure on `assignOrgAdmin` (AdminPage.jsx:15-19): an explicit mandate-#5 criterion ("assignOrgAdmin failures surface as errors, not silent success") is not met, and it doubles as an a11y miss (no announced error on the assign form). Fix: wrap `handleAssign` in try/catch and render a `role="alert"` in the row on failure (mirror `handleCreate`). The OrgLauncher `role="alert"` LOW should be fixed alongside it.
