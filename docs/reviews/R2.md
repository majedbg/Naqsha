# R2 — Integration + Security Review (READ-ONLY)

Reviewer: R2 (adversarial integration + security). Scope: `src/lib/org/*` service layer + cross-check vs spec `docs/org-admin-mvp.md` and migration `20250101000004_org_admin.sql`.

Live-RLS suite (rls.org.test.js + submissionService.test.js + platformService.test.js) was confirmed 30/30 PASS serially before this review.

## Findings

### 1. `submissionService.js` (+ .test.js) — SNAPSHOT IMMUTABILITY

Verified SAFE:
- `createSubmission` (lines 22–36) persists every snapshot field the spec §4 requires: `width_mm`, `height_mm`, `ops`, `material_label`, `svg_path`, plus `org_id`/`submitted_by`/`org_material_id`/`source`/`design_id`/`name`/`notes`. Good.
- `status: 'pending'` is **hard-coded last** in the payload literal (line 35). Because `status` is not a destructured input and is written as a literal AFTER no spread of caller input, a caller **cannot smuggle a status** — there is no `...rest` and no `status` parameter. Snapshot-immutability-on-create holds. SAFE.
- `markStatus` (lines 67–81) only ever writes `{ status }` and conditionally `cut_at` (line 70–72). It does NOT touch `width_mm/height_mm/ops/material_label/svg_path/org_id/submitted_by`. So the only mutable post-create fields through this service are `status` and `cut_at` — exactly the spec allowance. SAFE. The DB UPDATE policies (`submissions member update own` / `submissions admin update org`) further re-check `submitted_by`/`is_org_admin` and keep the `split_part(svg_path,'/',1)=org_id` invariant, so even a forged direct UPDATE cannot rebind svg_path/org. SAFE.
- All four read/write paths throw on `error` (lines 42, 53, 63, 79). No swallowing. SAFE.

LOW — `submissionService.js:67` `markStatus` — no status whitelist — *what:* `markStatus(id, status)` passes any string straight to the UPDATE; DB column has no CHECK constraint on `status` (migration line 80 only sets a default). A buggy caller could persist `status='garbage'`. *why:* low impact — it is not a privilege-escalation vector (RLS still gates WHO can update), only a data-integrity nit; the aggregation UI keys off known statuses. *fix:* optional — assert `status ∈ {pending,cut,rejected,canceled}` in the service or add a DB CHECK. Non-blocking.

MEDIUM — `submissionService.js:67` `markStatus` + migration `submissions member update own` (line 234) — a MEMBER can set `status='cut'` on their own row — *what:* `markStatus` has no role check, and the member-own UPDATE policy grants UPDATE on the member's own submission with NO column restriction. So a member can call `markStatus(theirOwnId, 'cut')` (even setting `cut_at`) and inject a "cut" job into the admin's queue / aggregation. *why:* spec §4 phrasing is ambiguous — "member reads/writes **own**" vs "**admin** … updates `status`/`cut_at`." If `cut` is an operator-only transition (likely — `cut_at` is a physical-machine fact), this is a real integrity hole; if members may self-cancel, the policy is fine for `canceled` but still over-broad for `cut`. *fix:* if `cut` is operator-only, restrict the member-update policy (or `markStatus`) so members may only set `status='canceled'`, leaving `cut`/`rejected` to the admin-update policy. Confirm intent with spec owner. MEDIUM.

NIT — `submissionService.js:67` `markStatus` — no org/ownership argument — relies entirely on RLS to scope which row a caller may flip. That is the correct boundary (RLS is the real gate), so this is acceptable by design; noting only that the service itself is identity-agnostic and trusts the authenticated client. SAFE-by-design.

Mock-vs-live note: the live smoke (test.js lines 133–218) genuinely exercises real RLS — member creates via `withUser({sub:memberId})`, admin sees through `submissions admin read org`, and a **peer member's `listForOrg` returns []** proving member-isolation is enforced at the DB, not the service (listForOrg has NO submitted_by filter, lines 57–65, yet peer sees nothing → proves the `submissions member own` select policy). This is a real protection, not a mock artifact. The snapshot/`status='pending'` assertions (lines 51–67) run against the mock, but immutability is also code-structurally guaranteed (literal, no spread). SAFE.

### 2. `platformService.js` (+ .test.js) — PLATFORM-ADMIN SECURITY

Verified SAFE:
- `createOrg` / `assignOrgAdmin` (lines 3–14, 39–55) contain **no client-side admin gate** — and that is correct per spec §3 ("the RLS policy — not the UI — is the real boundary"). The real protection is the DB policy `orgs insert platform … with check (is_platform_admin())` (migration line 180–182) and, for `assignOrgAdmin`, `org_members admin all … with check (is_org_admin(org_id))` (migration line 194–197). A non-platform / non-admin caller's INSERT is rejected by Postgres RLS regardless of what the client believes.
- Both functions throw on `error` (lines 12, 53). A denied insert surfaces as a thrown RLS error, never a silent success. SAFE.
- `is_platform_admin()` ANDs `jwt_email_verified()` (migration line 135) — so a forged/unverified `email` claim cannot become platform admin even if the email string matches the allowlist. The live smoke (test.js 158–162) sets `email_verified: true` to pass, confirming the verified-email gate is load-bearing. SAFE.
- `isPlatformAdmin()` (lines 16–29) is a **client convenience flag for hiding the Admin tab**, not a security boundary. Correctly filters `.eq('email', user.email)` (line 25) so it mirrors the `platform_admins read own` RLS policy and does not give a false-positive from an unfiltered select. Throws on error. SAFE — but it must never be the only gate (it isn't; RLS backs it).

LIVE-vs-MOCK — this is the key cross-check the brief asked for:
- The live smoke (test.js 140–192) **genuinely proves platform-only INSERT**: the verified platform-admin (`majed.bg@gmail.com`, `email_verified:true`) successfully creates an org, then a **different verified non-admin user** (`normal-…@example.com`) attempts `createOrg` and the test asserts `.rejects.toBeTruthy()` (line 186–188). Because `createOrg` throws on `error`, the rejection is the RLS denial — this proves the WITH CHECK is enforced at the DB for a real non-admin token. NOT a mock artifact. SAFE.
- Gap (LOW): the live smoke does NOT cover `assignOrgAdmin` against RLS — there is no live test that a non-admin is denied inserting an `org_members` admin row. The DB policy (`org_members admin all` with check) covers it, and rls.org.test.js may cover org_members generally, but `assignOrgAdmin`'s specific path is mock-only. See cross-check section.

LOW — live smoke does not assert email-UNVERIFIED denial — *what:* every live actor sets `email_verified:true`; no test proves an allowlisted-email-but-UNVERIFIED token is denied platform-admin. *why:* the unverified→denied path (the actual invite-hijack defense, spec §3) is therefore proven only by reading `is_platform_admin()`'s `and jwt_email_verified()` clause, not by a live assertion. *fix:* add one live case with `email:'majed.bg@gmail.com', email_verified:false` expecting `createOrg` to reject. Non-blocking (the SQL clause is present and correct).

### 3. `membershipService.js` — on-delete invariant + claimOnLogin

Verified SAFE:
- `removeMember` (lines 54–62) issues **exactly one DELETE against `org_members`** and nothing else. ZERO writes against `submissions`. The on-delete invariant (spec §3: "Removing a member does NOT cascade-delete their submissions") is preserved at the service layer. The DB FK is `submissions.submitted_by → profiles(id) on delete cascade` (migration line 69) — deleting an `org_members` row does not delete the `profiles` row, so no cascade reaches submissions either. SAFE at both layers.
- `addMemberByEmail` (3–18) writes email-first row with `status:'invited'`, `user_id` left null — matches claim-on-login model. Throws on error. SAFE.
- `editMember` (42–52): passes an arbitrary `patch` to UPDATE. See MEDIUM below.

MEDIUM — `membershipService.js:20-30` `claimOnLogin` — service path bypasses the verified-email security helper — *what:* the service implements claim as a plain client UPDATE `set user_id, status='active' where email=? and status='invited'`. This is NOT the migration's `claim_memberships()` SECURITY DEFINER RPC (migration 141–164), which is the path that enforces `jwt_email_verified()` (the invite-hijack defense, spec §3). *why:* a direct client UPDATE on `org_members` is gated by RLS. The only UPDATE policies are `org_members admin all` (requires `is_org_admin(org_id)`) and there is **no member-self-UPDATE policy** — `org_members read own` is SELECT-only (migration 198–200). So a normal logged-in user calling `claimOnLogin` will have their UPDATE silently match **zero rows** under RLS (or error), meaning **this function cannot actually claim a membership for a non-admin user**. It is either dead/ineffective for the real flow OR, if ever invoked by a service-role/elevated client, it would skip the `jwt_email_verified()` gate entirely and allow claiming by unverified email. *fix:* the real claim flow must call the `claim_memberships()` RPC (`supabase.rpc('claim_memberships')`), not this UPDATE. Confirm which path the login wiring uses; if `claimOnLogin` is wired into auth, it is a correctness BUG (claims won't happen) and a latent security gap (no email-verified check). Flag to R1/integration: verify login calls the RPC, and either delete this function or repoint it at the RPC.

LOW — `membershipService.js:42` `editMember` — unbounded `patch` — *what:* `editMember(id, patch)` forwards any object to UPDATE on `org_members`, including `org_id`, `email`, `user_id`, `status`. *why:* RLS (`org_members admin all` with check `is_org_admin(org_id)`) still confines this to an org the caller administers, so an admin can't escalate cross-org; but an admin could re-point `email`/`user_id` of a roster row or flip `status`. Within-org admin scope this is acceptable, but the open patch surface is worth narrowing. *fix:* whitelist editable keys (`is_admin`, `status`, `metadata`). Non-blocking.

NIT — `claimOnLogin`/`isOrgAdmin` filter on `status='invited'`/`is_admin=true` client-side (lines 28, 64–73). `isOrgAdmin` here duplicates the DB `is_org_admin()` helper as a client convenience; fine as a UI flag, not a boundary.

### 4. `uploadService.js` — storage path / org-prefix binding

Verified SAFE:
- Path is `${orgId}/${submissionId}.svg` (line 34) — first segment is `orgId`, exactly what the storage INSERT policy requires: `is_org_member(((storage.foldername(name))[1])::uuid)` (migration 270–276) AND what the submissions INSERT policy binds: `split_part(svg_path,'/',1) = org_id::text` (migration 232). Consistent. SAFE.
- Cross-org write defense is at the DB, correctly: even if a caller passes another org's `orgId` to build the path, the storage `insert owner` policy requires `is_org_member(orgId-from-path)` — a non-member is denied; and the submissions row insert requires both `is_org_member(org_id)` and the path↔org_id match. So an attacker cannot write under another org's prefix without active membership there, and cannot create a submission row pointing at a foreign prefix. The service trusting its `orgId` argument is fine because RLS is the real gate. SAFE.
- Input guard: rejects non-`<svg` strings (23–25) and >5MB by UTF-8 byte length (27–32). `owner = auth.uid()` is set by Supabase on upload, satisfying the storage read/update/delete owner policies. SAFE. Note the file's comment says input must be ALREADY-SANITIZED — sanitization is upstream's responsibility; this service only does a shape/size check, which is acceptable given the bucket is private and never rendered as active content to other tenants (admins download for cutting). Throws on error. SAFE.

### 5. `orgService.js` + `materialService.js` — error handling / null-masking

Verified SAFE:
- `getOrgBySlug` (orgService 3–14): `maybeSingle` + throw on `error`, returns `null` only for genuinely-zero-rows (unknown slug) — a real DB error still throws (line 12). No security-relevant masking. SAFE.
- `materialService` all four functions throw on `error` (lines 30, 42, 54 + listActive). `flattenOrgMaterial` explicitly picks `id: row.id` (org_materials.id) so the catalog material id can't clobber the toggle key (line 9) — good defensive denormalization. SAFE.
- `addOrgMaterial` spreads `...attrs` into the insert (line 36) but `org_id`/`material_id` are written explicitly BEFORE the spread, so an `attrs.org_id` could override them — see LOW. Writes are still RLS-gated by `org_materials write admin` (`is_org_admin(org_id)`), so cross-org abuse is blocked at the DB.

LOW — `materialService.js:36` `addOrgMaterial` — `...attrs` spread can override `org_id`/`material_id` — *what:* `{ org_id, material_id, ...attrs }` — because the spread is LAST, a caller passing `attrs={org_id:'other'}` overrides the explicit `org_id`. *why:* not exploitable cross-org: the RLS `with check (is_org_admin(org_id))` re-evaluates against the FINAL `org_id`, so an admin can only insert into orgs they administer anyway; but it's a footgun. *fix:* put explicit keys after the spread, or whitelist `attrs` to `{sheet_w_mm,sheet_h_mm,price,is_active}`. Non-blocking.

NOTE on `if (!supabase) return null/[]/false` pattern (all services): this masks "client not configured" as empty/null. In these services it is acceptable because (a) it only triggers when supabase is entirely absent (config-time, not auth-time), and (b) every genuine DB/RLS error still THROWS (the `if (error) throw error` is present in every path). So a security denial is never masked as "no data" — denials surface as thrown RLS errors. The only masking is the no-client case. SAFE.

### 6. Cross-check — do live smokes prove protections, or assert mock artifacts?

PROVEN LIVE (real DB RLS, not mock):
- **Member isolation** — submissionService.test.js peer-member `listForOrg` returns [] (lines 211–215) through real RLS despite no service-level filter → the `submissions member own` SELECT policy is genuinely enforced. REAL.
- **Platform-only org INSERT** — platformService.test.js: verified platform-admin inserts; verified non-admin's `createOrg` rejects (lines 180–188) → `orgs insert platform` WITH CHECK enforced at DB. REAL.
- **svg_path↔org_id binding** — submission live-create uses `${orgId}/...` (test 196); if it diverged the `split_part` check would deny. Exercised implicitly. REAL.
- rls.org.test.js (not re-read here, 30/30 confirmed) is the broad RLS suite covering the remaining table policies.

MOCK-ONLY (not proven by a live assertion):
- `assignOrgAdmin` non-admin denial — no live test; relies on `org_members admin all` WITH CHECK (covered structurally by migration + presumably rls.org.test.js, but not by this file's smoke). MEDIUM-of-coverage, not a code defect.
- Email-UNVERIFIED → platform-admin/claim denial — proven only by reading SQL (`and jwt_email_verified()` / claim_memberships guard), no live assertion. This is the spec §3 invite-hijack defense — its live coverage gap is worth closing.
- `removeMember` does-not-touch-submissions — asserted via mock/structure; trivially true from code (single DELETE on org_members). Acceptable.

CRITICAL INTEGRATION GAP (confirmed via grep): the migration ships a `claim_memberships()` SECURITY DEFINER RPC that is the ONLY correct, email-verified claim path — but **`grep` across `src/` finds NO caller of `claim_memberships` and no wiring of `membershipService.claimOnLogin` into any login flow**. So either (a) claim-on-login is not yet wired (an incomplete feature — invited members can never become active → can't submit), or (b) it will be wired to the WRONG path (`claimOnLogin`'s plain UPDATE, which RLS blocks for normal users and which skips `jwt_email_verified()`). Both outcomes are problems. This is the single most important finding for the integration owner.

---

## VERDICT: FINDINGS-MUST-FIX

No BLOCKER in the reviewed service code — the RLS boundary is sound and the live smokes prove the two headline protections (member isolation, platform-only org insert) against a real DB. However, the **claim-on-login flow is broken (correctness-first, not an exploitable bypass)**: the email-verified `claim_memberships()` RPC has no caller anywhere in `src/`, and `membershipService.claimOnLogin` — the only claim function in code — uses a plain client UPDATE that RLS denies for a normal authenticated user (no member-self-UPDATE policy exists; the pending row has `user_id IS NULL` so even read-own wouldn't match). Net effect: with the real anon/authenticated client this function is **non-functional** — invited members can never flip to `active`, so they can never submit. It is NOT an exploitable hole in the app as wired (the bypass of `jwt_email_verified()` would only materialize under a service-role client, which is not how these services run). MUST FIX before ship: wire login to `supabase.rpc('claim_memberships')` and remove/repoint `claimOnLogin`. Secondary (non-blocking): narrow `editMember`/`addOrgMaterial`/`addMemberByEmail` patch surfaces, add a `status` whitelist/CHECK on submissions, and add live coverage for unverified-email denial + `assignOrgAdmin` RLS.
