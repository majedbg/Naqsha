# R3 Review ‚Äî ¬ß7 Submit Experience + Submit‚ÜíSnapshot Path

Reviewer: R3 (adversarial, read-only). Scope: spec ¬ß7 submit experience and the
submit‚Üísnapshot write path. DEMO-GATE review.

Severity legend: BLOCKER / HIGH / MEDIUM / LOW / NIT.

## Findings

### A. `HoldToSubmitButton.jsx` ‚Äî UX / A11Y

- **MEDIUM ‚Äî HoldToSubmitButton.jsx:69 ‚Äî Space key fires submit instantly, bypassing the 2s friction.**
  `handleKeyDown` treats `' '` (Space) the same as Enter and calls `onConfirm` immediately.
  Spec ¬ß7 says only "`Enter` activates plainly" and explicitly rejects a 2s key-hold; it does
  not authorize Space as a second instant-submit path. Worse: a native `<button>` also emits a
  synthetic **click on Space keyup**. There is no `onClick` handler, so today this does not
  double-fire ‚Äî but it is fragile (adding onClick later silently double-submits real material).
  Why it matters: an extra, undocumented instant-commit keypath on a deliberate-friction control.
  Fix: handle only `Enter` in `keyDown`; if Space must work, call `e.preventDefault()` and confirm
  on `keyup` only, and never also wire `onClick`.

- **MEDIUM ‚Äî HoldToSubmitButton.jsx:80,86 ‚Äî No accessible reason for the disabled state.**
  When `disabled`, the button exposes no `aria-disabled` description / `aria-describedby` pointing
  at the completeness-gate reason. Spec ¬ß7 asks for "the disabled-reason via `aria`". A screen-reader
  user hears only "Hold to Submit, dimmed" with no "why". (The reason text lives in SubmitForm; it is
  not associated with this control.) Fix: accept a `reason`/`describedById` prop and set
  `aria-describedby` (and/or `aria-disabled` + visually-hidden text) when disabled.

- **LOW ‚Äî HoldToSubmitButton.jsx:38-41 ‚Äî touch‚Üímouse de-dupe flag can swallow a later genuine mousedown.**
  `touchedRef` is set true on every touch tap but only reset inside `startHold`. On a pure-touch tap
  no `mousedown` follows, so the flag stays armed; on a hybrid device a subsequent real
  `mousedown` is silently swallowed once (no hold starts). The double-fire guard itself works for the
  immediate synthesized event. Fix: clear `touchedRef` on a short timeout after touch, or key it to a
  timestamp window rather than a sticky boolean.

- **LOW ‚Äî HoldToSubmitButton.jsx:114 ‚Äî Label always reads "Hold to Submit" even on touch/keyboard where it is a plain tap.**
  On mobile and via keyboard the control submits on a single action, but the visible/accessible name
  still says "Hold to Submit", which is misleading on those modalities (minor; spec accepts the
  single shared label). Note only.

- **NIT ‚Äî HoldToSubmitButton.jsx:91-113 ‚Äî `aria-valuenow` only present in reducedMotion branch.**
  In the default (motion) branch the fill is `aria-hidden` with no progressbar semantics, so AT users
  on a non-reduced-motion machine get no progress feedback during the 2s hold. Acceptable per spec
  (visual gradient is the affordance) but worth noting.

**Verified SAFE (HoldToSubmitButton):** early mouse release truly resets (cancelHold clears both
timers and zeroes progress; test at :31 proves a later full hold still fires ‚Äî no armed-state leak);
`disabled` blocks mousedown/touch/keydown paths (guards at :36/:61/:67, test at :52); touch tap fires
once and the trailing synthesized mousedown is de-duped for the immediate event (:38, test at :65);
reduced-motion still enforces the full 2s (test at :87) and renders `role="progressbar"` with
valuemin/max/now. Timers are cleared on fire (:48). NOTE: no cleanup on unmount (no useEffect return)
‚Äî if the form unmounts mid-hold the timeout could still fire `onConfirm` on an unmounted tree; see
SubmitForm review for whether that is reachable.

### B. UX state machine ‚Äî `src/components/org/SubmitForm.jsx` (+ `.test.jsx`)

- **LOW ‚Äî SubmitForm.jsx:140-141 ‚Äî Hold-to-submit gate does NOT validate `orgId`/`userId` presence.**
  `handleConfirmSubmit` only guards on `ready` (material/size/ops) and `submitting`. If `useOrg`/
  `useAuth` haven't resolved when the user reaches the review step, `orgId`/`userId` are `undefined`
  and the hold can enable + submit to `uploadSubmissionSvg({ orgId: undefined, ... })` ‚Üí
  path `undefined/<id>.svg`, which RLS will reject. Reachable only if context lags the upload
  pipeline (UploadPipeline must complete first, so usually resolved by then). Why it matters: an
  undefined-tenant write attempt with no surfaced error (see C). Fix: include `!orgId || !userId`
  in the gate reasons (or block render until both present).

- **LOW ‚Äî SubmitForm.jsx:318-323 / HoldToSubmitButton unmount ‚Äî stray `onConfirm` is only reachable via EXTERNAL unmount.**
  Part A flagged HoldToSubmitButton has no unmount cleanup. Tracing reachability here: during a
  mouse hold the button stays mounted; the normal fire‚Üí`setSubmitting`‚Üí`setDone`‚Üíparent-unmount
  sequence all happens AFTER `onConfirm`. The ONLY reachable stray fire is an external unmount
  inside the 2s window (route change / logout / parent re-render dropping the form). Narrow, not on
  the demo path, but real. Fix lives in HoldToSubmitButton (clear timers in a useEffect cleanup).

**Verified SAFE (state machine):**
- Two-stage completeness gate surfaces ALL unmet reasons at once (`gateReasons` accumulates
  material + size + one entry per untagged layer, :98-109; tests prove "Pick a material" :113,
  "Confirm size" :121, "Tag layer N" :136). `ready = gateReasons.length === 0` wires
  `disabled={!ready || submitting}` into HoldToSubmitButton (:319); test asserts `holdBtn.disabled`
  true when ungated (:118) ‚Äî button is truly un-startable until clear.
- Hold-to-Submit is HIDDEN entirely while editing (button is in the non-editing branch :301-333;
  `gateReasons` returns `[]` while editing :101 so no stray "ready"); test proves the button
  disappears on Edit (:98-111). Submitting-while-editing is therefore impossible ‚Äî there is no
  submit affordance in the edit branch (only Cancel/Save).
- Edit‚Üîread-only round-trip: `enterEdit` snapshots a DEEP copy of layers (`layers.map(l => ({...l}))`)
  + name + materialId + sizeConfirmed (:113-114). `cancelEdit` restores all four and clears the
  snapshot (:118-127); test proves name + material revert (:192-214). Op-revert on cancel is NOT
  directly tested but is correct by inspection (layers deep-copied in, restored on cancel). `saveEdit`
  keeps current state and drops the snapshot (:129-132); test proves Save round-trips name+material
  into the read-only card and clears the gate (:156-190).
- Geometry is non-editable: the only size control is a "Confirm size" button (:199-207) that flips
  `sizeConfirmed` true ‚Äî there is NO width/height input. Submit writes `draft.widthMm/heightMm`
  (the parsed values), which is the intended immutable-geometry behavior.
- No gate-bypass found: every `ready` path requires materialId set, sizeConfirmed true, and every
  layer op non-null. There is no code path that clears `gateReasons` without those holding (the
  only other `[]` return is the editing branch, where the button is hidden).

### C. Correctness ‚Äî submit‚Üísnapshot path

`SubmitForm.handleConfirmSubmit` (:140-170) + `OrgSubmitPage.jsx` + `submissionService.createSubmission` + `uploadService.uploadSubmissionSvg`.

- **HIGH ‚Äî SubmitForm.jsx:167-169 ‚Äî Silent error swallow: a live upload/create failure produces ZERO feedback (DEMO-GATE).**
  ```js
  } catch { setSubmitting(false); }
  ```
  No message, no error UI, no `console.error`, no re-throw. If `uploadSubmissionSvg` or
  `createSubmission` fails in the live demo (RLS misconfig, auth/session not ready, network blip,
  5MB/`<svg` validation throw), the presenter holds the full 2s and the button silently re-enables
  with no indication anything went wrong. A real failure becomes invisible ‚Äî the worst demo failure
  mode. The test suite ONLY mocks both as resolved (:217-219); the entire failure path is untested,
  so this is assumed-not-proven on the unhappy side. Fix: set an error state and render it; at
  minimum `console.error(err)` and surface a retry-able message near the button.

- **MEDIUM ‚Äî SubmitForm.jsx:144-164 ‚Äî Orphaned blob when create fails after upload (and re-orphans on every retry).**
  Order is upload-then-create (correct: upload at :145, create at :151; test asserts
  `invocationCallOrder` upload < create at :252-255). But if `uploadSubmissionSvg` SUCCEEDS and
  `createSubmission` THROWS, the catch swallows it and the blob at `${orgId}/${submissionId}.svg`
  is orphaned with no DB row referencing it. Worse: each retry calls `newSubmissionId()` fresh
  (:144), so every failed retry orphans ANOTHER file. Untested (no failure-path test). Data-hygiene
  issue, not a happy-path breaker; no cleanup/compensation exists. Fix: on create failure, attempt
  to remove the just-uploaded object (or reuse a stable submissionId across retries so a retry
  overwrites rather than orphans).

**Verified SAFE (correctness):**
- Upload-then-create ordering is correct and proven (test :252-255). If upload throws, create is
  skipped (it's the next `await` in the same try, after the throw) ‚Äî proven implicitly: create only
  runs on the resolved-upload path.
- Snapshot fidelity / no stale-after-edit: the persisted row carries EDITED local state, not the
  original draft, for the fields the user can change ‚Äî `orgMaterialId: materialId`,
  `materialLabel: materialLabel(selectedMaterial)`, `ops` from `layers` (live `l.op`), `name` from
  state (:150-163). Immutable/geometry fields come from `draft` (`widthMm`, `heightMm`, `source`,
  `designId`, `notes`, `svgClean`) which is correct (geometry is non-editable by design). The
  happy-path test proves the edited material flows through: pick `om1` in edit ‚Üí Save ‚Üí submit
  writes `orgMaterialId: 'om1'`, `materialLabel: '3mm Birch Ply'`, `widthMm: 100`, `heightMm: 50`,
  `ops` length 2, `svgPath: 'org1/uuid-1.svg'` (:264-272). `status: 'pending'` is set in
  `createSubmission` (submissionService.js:35), not droppable by the form.
- `svg_path` prefix: `uploadSubmissionSvg` builds `${orgId}/${submissionId}.svg`
  (uploadService.js:34) and RETURNS that path, which the form passes straight to `createSubmission`
  as `svgPath` (:145-159) ‚Üí persisted as `svg_path` (submissionService.js:29). First path segment
  === org_id, satisfying the storage + submissions RLS requirement (test asserts
  `svgPath === 'org1/uuid-1.svg'` :269).
- Client-generated `submissionId` vs DB `row.id` mismatch: INTENTIONAL and harmless. The filename
  UUID (`newSubmissionId`, :30-37) intentionally differs from the DB-generated `row.id`; nothing
  keys off `filename === row.id`. `svg_path` is persisted explicitly on the row, and RLS only
  requires the first path segment to equal `org_id` ‚Äî not the filename. The header comment (:28-29)
  documents this. SAFE (not a finding).
- OrgSubmitPage wiring: `orgId = org?.id` (useOrg) and `userId = user?.id` (useAuth) are passed to
  `SubmitForm`, and the SAME `orgId`/`userId` are passed to `<MySubmissions>` after submit
  (:43, :51-57). `onSubmitted` flips to the submitted view (:55); `onCancel`/`reset` returns to
  upload (:56, :23-26). No wiring bug ‚Äî MySubmissions gets the correct tenant/user. (Caveat: the
  optional-chaining means undefined context propagates silently rather than blocking ‚Äî see B LOW.)
