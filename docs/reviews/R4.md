# R4 Review — Phase 4 E2E + Admin Security (READ-ONLY, adversarial)

Reviewer: R4. Scope: full Phase 4 e2e (upload + in-app submit) + admin authorization.
Spec: `docs/org-admin-mvp.md` §5/§7/§8. Headline question: *Does the
upload→queue→aggregate→cut loop hold end-to-end with no broken hop, and is every
admin action authorization-gated?*

## E2E-UPLOAD

Traced the full upload loop by reading the wiring at every hop. The mechanical
loop holds: a live run uploads → queues → aggregates → downloads → flips status
to `cut` with no broken hop. One MEDIUM fidelity bug at the aggregate hop.

### Hop-by-hop (verified-SAFE)
1. **OrgSubmitPage** (`src/pages/org/OrgSubmitPage.jsx`) — 3-state machine
   (`upload`→`review`→`done`). Passes `orgId=org.id`, `userId=user.id`,
   `draft` to SubmitForm; `onSubmitted`→done; Cancel→reset. Contract OK.
2. **UploadPipeline** (`UploadPipeline.jsx`) — file→`sanitizeSvg`→`parseDimensions`
   →`extractOps({source:'upload'})`. Emits a `draft` of
   `{source:'upload', svgClean, ambiguous, ops, removed, widthMm, heightMm}`.
   Ambiguous dims pause for a confirm-size step before `onComplete`. Matches §5.
3. **SubmitForm** (`SubmitForm.jsx`) — gate (`gateReasons`: material AND size
   confirmed AND every layer op-tagged AND not editing AND orgId/userId present)
   → HoldToSubmitButton → `uploadSubmissionSvg` then `createSubmission`. Client
   generates the submission id so the SVG lands at `${orgId}/${id}.svg` BEFORE
   the row exists. Orphan-blob cleanup on row-insert failure (best-effort
   `removeSubmissionSvg`). Gate + hold + ordering all sound.
4. **uploadService.uploadSubmissionSvg** — path `${orgId}/${submissionId}.svg`,
   re-checks `<svg`-prefix + 5MB byte ceiling, `contentType image/svg+xml`. The
   `${orgId}/` prefix is exactly what storage-insert RLS + the submissions
   insert WITH CHECK (`split_part(svg_path,'/',1)=org_id::text`) require. OK.
5. **submissionService.createSubmission** — inserts snapshot
   (`org_id, submitted_by, org_material_id, material_label, source, design_id,
   svg_path, width_mm, height_mm, ops, name, notes, status:'pending'`). Field
   names match the schema in §4. `ops` stored as `[{key,label,op}]` (mapped in
   SubmitForm:158). OK at this hop.
6. **MySubmissions** (`MySubmissions.jsx`) → `listMine(orgId,userId)` filters
   `org_id` + `submitted_by`; RLS double-scopes to own rows. New job shows
   Pending. OK.
7. **OrgAdminPage** → `AdminQueue` → `listForOrg(orgId)`, filters `status==='pending'`
   by `org_material_id`; selection lifted to OrgAdminPage; single-material guard
   derives one `sheet` from the shared `org_material`'s `sheet_w_mm/sheet_h_mm`.
   OK.
8. **AggregatePanel** — per selected row: `loadSubmissionSvg(sub.svg_path)`
   (download from private bucket) → `sanitizeSvg` (re-sanitize on render, §12)
   → `gridPlace(pieces, sheet)` (throws PIECE_TOO_LARGE/INVALID_* before any
   download/markStatus, so no half-commit) → `composeSheet` per sheet →
   `triggerDownload` → `markStatus(id,'cut')` for every placed id → `onCut`.
   svg_path round-trips correctly (upload writes `<orgId>/<id>.svg`, loader
   reads it back). Status pending→cut completes.

### Finding
- **MEDIUM — `src/lib/aggregate/composeSheet.js:30` (`processOf`) vs
  `src/components/org/SubmitForm.jsx:158` — op field-name mismatch — the
  member's op tags never reach the exported sheet.**
  Submissions persist `ops` as `[{key,label,op}]`. `composeSheet.opAttrs`/
  `processOf` read `op?.process` (object) or a bare string — neither matches the
  stored `{...,op}` shape. Result: `processOf` returns `undefined` for every
  piece, so each `<g>` is emitted with `data-op="undefined"` and the
  `OP_STROKE.cut` (red) fallback group-stroke, regardless of the score/engrave
  the member tagged. The §5 promise "ops normalized from tags" is not honored in
  the artifact.
  - Precise impact (not overstated): the red is a *group-level* `stroke`. For the
    **upload path**, inner elements carry their own stroke colors by construction
    (that is what `extractUpload` keyed ops on), and a child stroke overrides an
    inherited group stroke — so upload pieces export with their *original* colors,
    NOT all-red. The defect is that the cut/score/engrave normalization the spec
    calls for silently does nothing, and `data-op` is `"undefined"` on every group.
    For the **design path** (whose inner geometry may have no own stroke) the
    fallback red would actually paint through — but that path sits behind the
    known unbuilt studio adapter seam (see E2E-INAPP), so rate that consequence
    there.
  - Why not higher: the loop completes mechanically (SVG downloads, status flips
    to cut). This is an artifact-fidelity bug, not a hop that fails a live run.
  - Fix (one line): align the contract — either have `composeSheet.processOf`
    read `.op` (`op?.op ?? op?.process`), or map `{op}`→`{process}` at the
    AggregatePanel boundary when building `sheetPieces`.

### Known simplification (not a new finding)
- **composeSheet takes `ops[0]` for the whole `<g>`** — one piece = one op. The
  upload path can tag multiple ops per piece (one per stroke color), so a
  multi-op piece collapses to a single group-level op. Pre-existing modeling
  simplification, called out for completeness; orthogonal to the field mismatch.

## E2E-INAPP

`src/components/org/SubmitToOrg.jsx` builds a draft once on open:
`exportSvg()` → `sanitizeSvg` → `parseDimensions` (exact, `ambiguous:false`) →
`extractOps({source:'design'})` → draft `{source:'design', svgClean, widthMm,
heightMm, ambiguous:false, ops, designId, name}` → renders the SAME `SubmitForm`
as the upload path. Given a *correct* `exportSvg`, the component path is sound.

### Component path — verified-SAFE (given a proper exportSvg)
- `extractOps(source:'design')` (`src/lib/svg/extractOps.js:25`) keys ops off
  `g[data-role]` groups, mapping role→op via `ROLE_OPS{cut,score,engrave}` —
  exactly the contract a correct adapter must emit (§16b: in-app ops from layer
  roles, NOT stroke color). `extractDesign` matches what a proper adapter must
  produce.
- Downstream of the draft, the design path reuses SubmitForm verbatim, so the
  upload-path hop verification (gate→hold→upload→createSubmission) applies
  identically. `source:'design'` and `designId` flow through to
  `createSubmission` (`design_id … on delete set null`, provenance only). OK.

### KNOWN SEAM (documented; not a component defect)
- **HIGH-as-a-gap-but-known — the studio "Submit to org" button + the
  `buildAllLayersSVG`→`data-role` adapter are NOT wired.** Evidence:
  - `SubmitToOrg` is imported nowhere outside its own test (grep across `src/`).
    There is no studio entry point that mounts it with an `exportSvg`.
  - The real exporter `src/lib/svgExport.js:buildAllLayersSVG` emits
    `<g id="…" opacity="…">` with **no `data-role`** — process/op lives only in
    a metadata comment (line 159–165), not on the group. So feeding the current
    real export straight into `extractOps({source:'design'})` returns `[]`
    (no `g[data-role]` matches) → zero op rows → the completeness gate can never
    pass (every "Tag layer N" reason, but no layers to tag). The adapter that
    maps each layer's `operation.process`/`penSlot`→`data-role` is the missing
    piece.
  - This is exactly the seam the spec build-sequencing (§6 item 3) and the
    integrator note in `SubmitToOrg.test.jsx` flag. Per the review brief it is a
    KNOWN seam — rated, not treated as a broken component.
- **Interaction with the E2E-UPLOAD MEDIUM:** if/when the adapter IS built, the
  composeSheet `op`-vs-`process` mismatch bites HARDER on the design path,
  because design-path inner geometry may carry no own stroke, so the erroneous
  red group-stroke fallback would actually paint through (vs upload pieces that
  keep their own colors). Fix the composeSheet contract before wiring the studio
  adapter, or the first real in-app aggregate ships all-red sheets.

## SECURITY-ADMIN

Two-layer model holds: client gate + RLS-as-real-boundary. Every admin action is
authorization-gated, and a client-only bypass cannot escalate because RLS
re-checks `is_org_admin(org_id)` server-side. No findings — all verified-SAFE.

### AdminQueue — verified-SAFE
- `src/components/org/admin/AdminQueue.jsx:9` — the `listForOrg(orgId)` effect is
  guarded by `if (!isAdmin) return undefined;` and renders "Access denied —
  admins only" when `!isAdmin`. A non-admin never triggers the queue fetch.
- Defense-in-depth: even if the client gate were bypassed, `listForOrg`
  (`submissionService.js:57`) is a plain `select` over `submissions`. RLS policy
  `submissions member own` scopes a non-admin to `submitted_by = auth.uid()`;
  the `submissions admin read org` (`is_org_admin(org_id)`) policy only widens
  for actual admins. A non-admin calling `listForOrg` gets only their own rows,
  never the org-wide queue.

### AggregatePanel markStatus('cut') — verified-SAFE (admin-only at RLS)
- `AggregatePanel.jsx:56` calls `markStatus(id,'cut')`. OrgAdminPage only mounts
  AggregatePanel under `isAdmin && selected.length > 0` (line 113), so the
  component path is admin-gated client-side.
- RLS is the real boundary and confirms the R2 fix still holds: a member's
  UPDATE (`submissions member update own`, migration:234) has WITH CHECK
  `status in ('pending','canceled')` — a member literally cannot set `'cut'`
  (or `'rejected'`). Only `submissions admin update org` (`is_org_admin(org_id)`,
  migration:255) permits arbitrary status incl. `cut`. RLS is OR-of-permissive,
  so majed (admin AND member of itp-camp) can still cut; a plain member cannot.
  A non-admin invoking `markStatus(id,'cut')` directly is denied by RLS.

### OrgAdminPage — verified-SAFE (direct-nav protected)
- `OrgAdminPage.jsx:34` resolves `isOrgAdmin(orgId,userId)` async and renders a
  loading state until resolved; AdminQueue receives `isAdmin` and self-renders
  "Access denied" when false; AggregatePanel is gated behind `isAdmin`. A
  non-admin who direct-navs to `/o/:slug/admin` reaches the page shell but gets
  the access-denied queue and no aggregate panel, and any data call they could
  still force is RLS-scoped. No client-only authority.
- `isOrgAdmin` (`membershipService.js:76`) requires
  `status='active' AND is_admin=true` for the (org,user) pair — an invited or
  non-admin member resolves false.

### Cross-org isolation — verified-SAFE
- All admin helpers are per-org: `is_org_admin(org)` checks membership *for that
  specific org*. `listForOrg`/`markStatus`/aggregate writes all flow through
  `is_org_admin(org_id)` keyed on the row's own `org_id`. An admin of org A
  calling `markStatus` on an org-B submission fails the `submissions admin update
  org` WITH CHECK (`is_org_admin(B)` is false for an A-only admin). Reading org-B
  rows fails `submissions admin read org` likewise. No cross-org authority.

### Storage boundary — verified-SAFE
- Bucket `submissions` is private. Insert RLS (migration:277) requires
  `owner=auth.uid() AND is_org_member(folder_org_id)`; read requires owner OR
  `is_org_admin(folder_org_id)`. The `${orgId}/` path prefix is bound to the row
  via the submissions insert WITH CHECK `split_part(svg_path,'/',1)=org_id::text`
  (migration:232) — an attacker-chosen prefix can't divert a row to another org's
  folder. R1/R2/R2.5 boundaries intact under the Phase 4 components.

### Summary
Affirmative answer to the headline's admin half: **every admin action
(queue read, status→cut, aggregate write) is authorization-gated, with RLS — not
the UI — as the enforced boundary, and cross-org action is impossible.**

## Verdict: FINDINGS-MUST-FIX

**Headline:** The upload→queue→aggregate→cut loop holds end-to-end with **no
broken hop** — a live run completes mechanically — and **every admin action is
authorization-gated at the RLS layer, with no cross-org authority**. But there is
one MEDIUM artifact-fidelity defect that must be fixed before the aggregate
output is trustworthy.

### Must-fix
- **MEDIUM — `composeSheet.processOf` (`src/lib/aggregate/composeSheet.js:30`) vs
  `SubmitForm.jsx:158`** — stored ops are `{key,label,op}` but composeSheet reads
  `op?.process`/bare-string, so `data-op="undefined"` on every group and the
  member's cut/score/engrave tags are NOT normalized into the exported sheet
  (§5 "ops normalized from tags" unmet). Upload pieces keep their own colors
  (child stroke overrides group stroke); the design path — once its adapter is
  wired — would paint the erroneous red fallback through. One-line fix: align the
  field (`op?.op ?? op?.process`) or map at the AggregatePanel boundary. **Fix
  this before wiring the studio adapter.**

### Known seam (rated, not a blocker per brief)
- Studio "Submit to org" button + `buildAllLayersSVG`→`data-role` adapter are
  unbuilt; `extractOps({source:'design'})` would return `[]` against the current
  real exporter. Component path is correct given a proper `exportSvg`; only the
  wiring is missing (spec §6 item 3).

### Verified-SAFE (banked)
- Upload path: every hop's data contract holds (field names, `${orgId}/` prefix,
  dims/ops snapshot, status pending→cut).
- Admin security: AdminQueue gated + RLS-scoped; `markStatus('cut')` admin-only
  at RLS (members capped to pending/canceled); OrgAdminPage direct-nav protected;

- Gate: build green; test suite green modulo one unrelated CirclePacking timeout
