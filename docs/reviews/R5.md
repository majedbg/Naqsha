## NON-NEGOTIABLES (spec §3)

### 1. Email-verified gate — VERIFIED
- `is_platform_admin()` ANDs `public.jwt_email_verified()` — `supabase/migrations/20250101000004_org_admin.sql:129-136`. An unverified email cannot satisfy platform-admin even if seeded in `platform_admins`.
- `claim_memberships()` returns early (`return;`) when `not jwt_email_verified()` before any `update` — `:152-154`. Unverified login cannot flip `org_members.user_id`/`status` or claim `platform_admins.user_id`. Invite-hijack vector closed.
- `jwt_email_verified()` reads both top-level and `user_metadata.email_verified`, defaulting `false` — `:94-101` (defensive across synthetic/real tokens).
- `config.toml:209` `[auth.email] enable_confirmations = true`. (Note: `:244 enable_confirmations = false` is `[auth.sms]`, irrelevant to email auth.)

### 2. on-delete rules — VERIFIED
- `submissions.design_id ... on delete set null` — `:73` (snapshot is source of truth; deleting personal design keeps the job).
- `submissions.org_material_id ... on delete restrict` — `:70`; `material_label text` denormalized snapshot — `:71`.
- No FK from `submissions` to `org_members`; deleting an `org_members` row cannot cascade to submissions. (`submitted_by` FKs `profiles`, not the roster.) Job survives member removal — VERIFIED.

### 3. RLS isolation — VERIFIED
- Member reads own only: `submissions member own ... using (submitted_by = auth.uid())` — `:222-224`. No member-side cross-member SELECT policy exists.
- Admin sees all org: `submissions admin read org ... using (is_org_admin(org_id))` — `:252-254`.
- Member status cap: `submissions member update own` with-check pins `status in ('pending','canceled')` — `:234-248`; `cut`/`rejected` reachable only via the admin update policy `:255-258` (RLS is OR-of-permissive).
- Cross-org denied: every policy keys on `is_org_member/is_org_admin(org_id)`; insert binds `is_org_member(org_id)` and `split_part(svg_path,'/',1) = org_id::text` (`:225-233`) preventing path/row org divergence (R1 fix). Update keeps the same path binding `:234-248`.
- Orgs INSERT/UPDATE platform-only — `:180-186`.

### 4. Sanitization — VERIFIED
- `sanitizeSvg` (`src/lib/svg/sanitizeSvg.js`): DOMPurify SVG profile strips `<script>`, event handlers, `<foreignObject>` (USE_PROFILES svg). `FORBID_TAGS:['style']` + `FORBID_ATTR:['style']` (`:66-67`) kill CSS `url()` exfil. `afterSanitizeAttributes` hook strips any non-`#fragment` href/xlink:href/src — neutralizes remote/`//host`/`data:`/`javascript:` while preserving same-document `#id` (`:23-41`). `<use>` re-allowed but href-constrained.
- Test coverage (`sanitizeSvg.test.js`): script, onload/onclick, foreignObject, `#fragment` preserved, `javascript:` URI, `data:` URI, `@import url(remote)`, `<style>` CSS url() remote, inline `style=""` url() remote, and a combined-vector case. All vectors in §3 covered.
- **Sanitize-on-upload-BEFORE-store (the foundational §12 guarantee): VERIFIED.** `UploadPipeline.jsx:21` runs `sanitizeSvg(raw)` first; dims/ops are parsed from the `clean` string and only `svgClean` propagates via `onComplete` (`:24-39`). `SubmitForm.jsx:153-156` uploads `svgString: draft.svgClean` — the sanitized string, never `raw` — so only sanitized SVG ever reaches the private bucket (`uploadService.uploadSubmissionSvg`). The in-page preview also renders `draft.svgClean` (`SubmitForm.jsx:211 dangerouslySetInnerHTML`), not raw. `MySubmissions.jsx` is a list view that does NOT render stored SVG inline, so there is no unsanitized member-side render path.
- Re-sanitize on aggregate render: `AggregatePanel.jsx:37` calls `sanitizeSvg(raw)` after `loadSubmissionSvg(svg_path)` per §12 (defense-in-depth, in case a pre-fix object predates store-time sanitize). VERIFIED.

### Notes (non-blocking, no finding)
- **Member status revert semantics:** `submissions member update own` `USING` is `submitted_by=auth.uid()` only; the `pending/canceled` cap is in `WITH CHECK`. A member can therefore revert an admin-set `cut`→`pending` (target is within the allowed set). The literal non-negotiable — a member never *sets* `cut`/`rejected` — holds. This is an accepted semantic quirk, not a violation. (RLS suite green; if it does not assert "member cannot regress cut→pending," that specific edge is unasserted-but-permitted-by-design.)
- **Test exit code:** the `exit 0` on the full unit run came from the bash wrapper's trailing `echo`, not vitest's own code (vitest exits non-zero on failure). The authoritative evidence that the 13 failures are flakes is the clean isolated re-run (8/8 files green), not that `0`.
